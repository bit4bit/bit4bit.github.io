¿Qué? ¿Cómo? ¿Quién?

Una opinión para orientarnos en el diseño del sistema.

¿Quién? es responsable de ¿Qué?.

Siendo el ¿Qué? la abstracción del ¿Cómo?.

He observado que cuando nos estamos iniciando el desarrollo de software nos centramos principalmente en el ¿Cómo? posteriormente y chapaleando empezamos a entender el ¿Qué y cuando finalmente llegamos al ¿Quién? volvemos a empezar desde el ¿Cómo?. Con el tiempo empezamos a quedarnos prinicipalmente entre el ¿Quién? y el ¿Qué? con escapes furtivos al ¿Cómo?. ¿Quién? es responsable de la lógica empresarial y ¿Qué? actividades puede realizar, el ¿Quién? nos permite orientador en idenficar los actores en la coleccion de elementos que son de utilidad para expresar las solución que es estamos implementado para el contexto y el momento actual.

Tenemos ¿Quién? a diferentes niveles de composición:

  • nombre de variable
  • nombre de metodo
  • nombre de clase
  • nombre de módulo
  • nombre de archivo

Ahora siguiendo el principio de buscar la estabilidad, miremos los diferentes nivels:

desde el mas propenso a la estabilidad al menos propenso

  • nombre de módulo
  • nombre de clase
  • nombre de metodo
  • nombre de archivo

nombre de archivo, lenguajes como Elixir, Ruby, Python no requieren que el nombre de archivo tenga relación con los elementos que estan contenidos en el, osea que facilmente un archivo se puede renombrar y el programa no se ve afectado.

nombre de módulo, lenguaje como Elixir, Ruby, Python no asocian el nombre del modulo con el archivo, si la asociacion entre modulos no es dinamica facilmente se puede reajustar.

!!! sigo sin lograr orientar el post naa

¿Quién? se hace responsable del ¿Qué? por medio del ¿Cómo?, sería la idea principal.

Vamos a revisar un poco de código para identificar abstracciones usando las preguntas guías.

#include <stdio.h>

int main() {
    // ¿Qué? imprimir en consola
    // ¿Cómo? ni idea
    // ¿Quién? `stdio.h`
    puts("oeee");
    return 0;
}

El ejemplo anterior podemos identificar ¿Qué? puts, ¿Cómo? es lo que buscamos evitar con la abstracción del ¿Qué? y el ¿Quién? sabemos que al includer stdio.h tenemos acceso a la función exportada puts, ahora usar los nombres de archivos que tanto conocimiento nos da del ¿Qué?.

Otro ejemplo

io:format('oeee').

Ahora el ¿Qué? format, ¿Cómo? ni idea, ¿Quién? io

otro ejemplo

IO.puts "oeee"

Ahora el ¿Qué? puts, ¿Cómo? ni idea, ¿Quién? IO, un modulo que tanto conocimiento nos da?

Como podemos entre mas concreto sea el ¿Quién? podemos hacernos una idea rapidamente de que ¿Qué? nos puede ofrecer.

La orientación a objetos nos fácilita generar una conversacion entre ¿Quiénes? para entre si ofrecer el ¿Qué?.

uri = URI('https://www.gnu.org')
req = Net::HTTP::Get.new(uri)
res = Net::HTTP.start(uri.hostname, uri.port) {|http|
  http.request(req)
}

acoplamiento de:

  • el ¿Quién? en un nombre de variable?
  • el ¿Quién? en un nombre de metodo?
  • el ¿Quién? en un nombre de clase?
  • el ¿Quién? en un nombre de archivo?
  • el ¿Quién? en un nombre de modulo? erlang

¿Quién? es una abstración del ¿Qué? que este a su vez es una abstracción al ¿Cómo?
